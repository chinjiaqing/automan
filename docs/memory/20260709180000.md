# 2026-07-09 工作总结 — OCR 全栈实现 + 性能优化

## 完成事项

1. **OCR 功能全栈实现**
   - `ocr.py` — Python OCR 脚本（RapidOCR + ONNX Runtime），支持 getWords / findStr
   - `ocr.ts` — TypeScript 封装层，`getWords()` + `findStr()` 两个公开 API
   - `shared/types.ts` — OcrColor / OcrWord / OcrMatch / GetWordsRequest / FindStrRequest 等类型

   - `OcrPanel.vue` — 识字/找字切换面板，支持颜色过滤、相似度调节

2. **持久化 Python 工作进程（PythonWorker）**
   - 新增 `PythonWorker` 类（python-helper.ts），进程常驻、模型只加载一次
   - ocr.py 新增 `--worker` 模式，逐行 JSON 协议，预热后输出 `ready` 信号
   - 性能提升：warm 调用 ~2.1s（vs 之前 ~3.5s），提升约 40%

3. **编码修复**
   - Windows 上 Python 子进程输出乱码 → spawn 时设置 `PYTHONIOENCODING=utf-8`
   - 同时修复一次性模式（runPythonScript）和 worker 模式（PythonWorker）

4. **找字匹配算法升级**
   - 新增 `levenshtein_ratio()` — 基于编辑距离计算字符串相似度
   - 新增 `sliding_window_match()` — 滑动窗口模糊子串匹配，容忍 OCR 单字误识别
   - `findStr` 返回 `allWords` 字段供调试

5. **文档更新**
   - 开发前必读：§15 新增编码规范、§16 OCR 与图像识别、§17 分发与运行时

## 关键决策

- **RapidOCR** 而非 PaddleOCR / EasyOCR：轻量、中文好、ONNX Runtime 跨平台
- **PythonWorker** 而非每次 spawn：模型加载开销从每次 3.5s 降到首次 3.4s + 后续 2.1s
- **Levenshtein 编辑距离** 而非简单字符比对：应对 OCR 单字误识别（如 "雷电" → "雷龟"）
- **PYTHONIOENCODING=utf-8**：Windows 默认 GBK 编码导致中文乱码的标准解决方案
