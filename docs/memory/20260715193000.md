# 工作总结：设备级生命周期与定时触发解耦（inactive 状态引入）

## 一、本次完成的事项

修复"定时 workflow 达标后误杀设备级任务与兄弟定时"的 Bug，并把"设备级任务"与"workflow 激活态"彻底解耦。

### 复现的 Bug
单设备 + 单脚本，定时模式设两个时间点（14:53 / 14:54），完成条件均为"成功 1 次后停止"。14:53 达标后：任务标记完成 → 设备级任务被自动结束 → 14:54 定时被销毁 → 设备会话被销毁。

### 根因
`WorkflowService.syncDispatcher` 的"全部 actor completed/cancelled"分支执行整套拆除：
- `cronManager.unregister(deviceId, workflowId)` 按 `deviceId:workflowId` 整组删除同一 workflow 的**全部定时 job（含 14:54）**；
- `dispatcher.forceStop()` 端掉截图循环；
- `session.destroy()` 销毁设备会话；
- `actors.delete()` 删除 actor。

与"cron 生命周期由设备级 start/stop 控制、completed 不触发销毁"的设计注释自相矛盾。

## 二、需求重定义（用户确认）

1. **设备级任务常驻**：无激活 workflow 时，截图循环 entry 与设备会话保留，仅暂停截图识别（`pause`），不销毁。设备级只由显式 `stopWorkflow` / `stopByDevice` 释放。
2. **定时 = 触发器 + 初始化器**：定时到达 = 未激活 → 激活 + 重置初始参数（计数清零、取消标记清除、session 变量清空、input/local 复位）。多定时点互不叠加。
3. **cron 每天重复、持久**：`cron.manager` 表达式本就是 `${minute} ${hour} * * *`，不应被自动清除。
4. 用户确认引入 `inactive` 状态（成本低、语义清晰）。

## 四、关键决策及原因

- **引入 `inactive` 而非复用 `completed`**：`completed` 之前既表示"立即模式真停"又被 `markPending` 当"新一轮起点"，语义混叠。拆出 `inactive` 后，"达标真停" vs "达标回等待"干净分开，UI 也能正确显示"等待定时"而非误显"已完成"。
- **session 变量随定时重置**：`sessionVars` 是 actor 私有字段，仅绑定 `device+workflow` 唯一实例，不跨 workflow/设备，故每次唤醒清空以实现"全新激活"。
- **设备级只由显式 stop 释放**：`stopWorkflow` 路径保持 `cronManager.unregister` + `session.destroy` 不变，符合"用户主动停止就释放设备"预期；自动路径不再销毁。
- **busy 时跳过唤醒**：契合"设备下 workflow 唯一、不叠加"，正在跑时定时信号不打断当前执行。

## 五、状态机（定时模式）

```
inactive ──(cron: 重置+唤醒)──> pending ──(截图)──> processing
   ▲                                                    │
   └──────────(达标: 回未激活, 等下次 cron)─────────────┘
                  (未达标: autoPending → pending)
```
