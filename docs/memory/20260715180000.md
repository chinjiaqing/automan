## 一、本次完成的事项

对应审查报告 `docs/.qoder/plans/Workflow_Fragments_流程审查_98639d47.md`（16 项问题），在引擎层一次性完成修复，并补修一处回归 bug：

1. **统一执行入口（最大架构问题 #11）**：将 `execute()` / `execLoop()` / `executeFragment()` 三处重复的节点分发逻辑抽取为私有方法 `executeNode()`，三处统一调用。新增节点类型只需改一处。
2. **功能缺陷修复**：循环体支持 `call`（#1）；片段支持嵌套 `call`（#3）；"片段 > 循环 > 调用片段"链路打通（#4）；`resolveRegion` 透传 `variables`（#5）；删除片段前检查引用（#9）；删除工作流前检查运行中（#10）；`category` 联合类型加入 `'fragment'`（#12）；截图调度器计数脱耦（#13）；移除死代码 `resolveConfig`（#14）；前端 `refResolver` 对齐 `{{var:name}}` / `{{scope.name}}`（#15）；`endSuccess`/`endFail` 在循环中正确向上传播（#2/#16）。
3. **变量/输出隔离与递归保护（#6/#8）**：`execCall` 使用独立 `ctx.outputs`；`MAX_CALL_DEPTH = 10` 防无限递归。
4. **补修回归 bug**：重构初期 `executeNode` 的 `return` 分支未携带 `returnValues`，导致 "return 嵌套在 loop 内（且位于片段中）" 时返回值丢失。修复为：在 return 节点处直接计算并透传 `returnValues`，`executeFragment` 优先使用透传值。

## 三、关键决策及原因

- **以"统一 executeNode"为重构核心**：审查指出三处重复是最大维护风险。统一后节点执行逻辑单点维护，且天然让 `call`/`loop`/`return` 在所有上下文（主流程/循环/片段）行为一致，连带解决 #1/#3/#4/#2/#16。
- **输出隔离用独立 `ctx.outputs` 而非共享写入**：避免片段内部节点 ID 与父工作流碰撞（#6），仅在 return/end 时按声明输出项合并。
- **return 值在 return 节点处解析并透传**：因为执行链中 `innerNode` 在嵌套场景会变为 loop 节点，无法在 `executeFragment` 层重新解析，故在源头（return 节点）就算好并随 `action` 上行。
- **截图调度器计数脱耦**：`syncDispatcher` 改为 pause/resume/forceStop，`start()`/`stop()` 的订阅计数仅保留给显式 `stopWorkflow` 路径，避免事件多次触发导致计数失准（#13）。

## 四、遗留问题 / 下一步计划

1. **UI 编辑器尚未开放片段内 `call`/`return`**：`web/src/views/FragmentView.vue` 的 `:exclude-types="['call','endSuccess','endFail','end','return']"` 仍在排除。引擎已具备嵌套能力，可视需求决定是否放开（放开需配套 `call` 节点片段选择/参数表单）。
2. **变量浅拷贝（#7）**：当前仅基本类型安全，未来支持对象/数组引用需升级为深拷贝或不可变更新。
3. **前端算术求值（#15 残留）**：后端 `resolveValue` 支持内嵌算术，前端 `resolveRefs` 暂不支持，仅影响预览显示，优先级 P3。
