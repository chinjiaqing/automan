# 坐标与尺寸问题分析文档

## 1. 系统坐标链路总览

系统中存在 **三种图像尺寸空间**，坐标在不同空间之间需要正确转换：

| 空间 | 说明 | 典型尺寸 |
|------|------|--------|
| **设备实际分辨率** (actualWidth × actualHeight) | 设备当前屏幕方向的像素尺寸 | 如 3168×1440（横屏）、1280×720 |
| **标准空间** (standardWidth × standardHeight) | 截图经 sharp resize 后的尺寸，最长边=1280 | 如 1280×581、1280×720 |
| **显示像素空间** (displayW × displayH) | 前端 `<img>` 实际渲染尺寸 | 随容器宽度自适应 |

### 坐标转换关系图

```
显示像素 ←→ 标准空间 ←→ 设备实际分辨率
 (CSS渲染)   (1280 max)     (设备原生)
           × scaleFactor
```

## 2. 各环节现状分析

### 2.1 Track 页面（CanvasView）— 框选与调试

**截图来源**：`POST /api/devices/screenshot`

**现状**：
- 该接口返回 **原始分辨率** 的完整 PNG（不压缩）
- `screenshot.width` / `screenshot.height` = 原始设备分辨率
- 没有 `originalWidth` / `originalHeight` 字段（与实时看板数据结构不同）

**框选坐标转换** (`useSelection.ts`)：
```ts
scaleX = screenshot.width / imgDisplaySize.width   // 原图宽 / 显示宽
scaleY = screenshot.height / imgDisplaySize.height // 原图高 / 显示高

selection = {
  x1: box.left * scaleX,   // 显示像素 → 原始分辨率
  y1: box.top * scaleY,
  x2: (box.left + box.width) * scaleX,
  y2: (box.top + box.height) * scaleY,
}
```

**结论**：Track 页面框选输出的坐标 = **原始设备分辨率**，这是正确的。

**存在的问题**：
1. **`saveSelection()` 裁切 Bug**：裁切代码使用 `selection` 坐标（原始分辨率）在压缩图（720px）上 `drawImage`，坐标体系不匹配，导致裁切位置完全错误
2. **Track 截图 API 与 Workflow 截图 API 数据结构不一致**：Track 返回 `{ width, height }` (原图尺寸)，Workflow 的 `ScreenshotDispatcher` 返回 `{ width, height, originalWidth, originalHeight }` (压缩尺寸 + 原图尺寸)

---

### 2.2 Workflow 参数填写

**现状**：
- Workflow 节点（识图、识字、点击等）的 `region` 参数约定为 **原始设备分辨率**
- 用户在 Track 页面框选后复制坐标 `[x1, y1, x2, y2]`，直接粘贴到 Workflow 节点配置

**存在的问题**：
1. **设备分辨率不一致风险**：如果用户在设备 A（1920×1080）上框选，但 Workflow 跑在设备 B（2560×1440）上，坐标完全不适用，系统无任何校验或提示
2. **无坐标空间标注**：前端 Workflow 配置面板未明确标注"坐标基于哪个分辨率"，用户容易混淆
3. **模板图片来源不明确**：Track 页面截图是原始分辨率，用户从截图上裁切的模板图片分辨率与 Workflow 运行时看到的压缩图分辨率不同

---

### 2.3 Track 页面回显（找图/识字结果）

**现状**：

- **找图 (FindPicPro)**：Track 页面发送原始分辨率截图 + 原始分辨率 region 到 `POST /api/devices/find-pic-pro`
- 返回的 `matches[].x / y` 是在 **发送图片空间**（即原始分辨率）中的坐标
- CanvasView 渲染覆盖层时：
  ```ts
  sx = imgDisplaySize.width / screenshot.width   // 显示宽 / 原图宽
  sy = imgDisplaySize.height / screenshot.height // 显示高 / 原图高
  overlay = { left: match.x * sx, top: match.y * sy, ... }
  ```
  这里 `screenshot.width` = 原图宽，所以转换正确

**存在的问题**：
1. **性能问题**：Track 页面每次截图传输完整原始分辨率 PNG（可能 3-5MB），带宽和延迟远高于压缩图
2. **找图在原始分辨率上执行**：SIFT 特征匹配在大图上更慢，而压缩到 720px 后匹配效果差异不大

---

### 2.4 Server 端实际位置计算（Workflow Engine）

**现状**（`engine.ts`）：

```ts
// 1. 用户参数（原始分辨率）→ 压缩图空间
scaleRegionToCompressed(region, ctx) {
  sx = ctx.screenshotWidth / ctx.originalWidth   // 如 720 / 1920 = 0.375
  return [region[0]*sx, region[1]*sy, region[2]*sx, region[3]*sy]
}

// 2. 在压缩图上执行 findPicPro / ocrFindStr（使用 compressedRegion）

// 3. 结果坐标（压缩图空间）→ 原始分辨率
scaleX = ctx.originalWidth / ctx.screenshotWidth   // 如 1920 / 720 = 2.667
matchX = Math.round(match.x * scaleX)
matchY = Math.round(match.y * scaleY)
```

**存在的问题**：
1. **整数舍入累积误差**：`scaleRegionToCompressed` 用 `Math.round`，结果缩放回来又用 `Math.round`，两次舍入可能导致 1-2px 偏差
2. **非等比缩放风险**：如果 `originalWidth/originalHeight` 比例与 `screenshotWidth/screenshotHeight` 比例不完全一致（理论上 sharp 等比缩放不会发生，但如果原始尺寸读取失败导致 fallback 则可能出问题）
3. **`[0,0,0,0]` 全图约定**：region 为 `[0,0,0,0]` 表示全图搜索，但这个"魔法值"缺乏类型安全，用户可能误填

---

### 2.5 日志页面回显（ExecutionViewer）

**现状**：

```ts
// 注解坐标基于原始分辨率
sx = displayW / screenshot.originalWidth   // 显示宽 / 原始宽
sy = displayH / screenshot.originalHeight  // 显示高 / 原始高

// 渲染 bbox/text/area/click 注解时乘 sx/sy
```

**结论**：日志页面回显逻辑正确，注解坐标明确基于原始分辨率。

**存在的问题**：
1. **bbox 固定 40px 方框**：`BOX_HALF = 20`，无论原始分辨率多高，显示框始终是 40px，在高 DPI 设备上可能太小难以辨识

## 3. 核心问题汇总

| # | 问题 | 严重程度 | 影响范围 |
|---|------|---------|---------|
| 1 | Track 页面 `saveSelection()` 裁切使用原始坐标在压缩图上操作 | **高** | Track 页面保存选区功能不可用 |
| 2 | Track 截图 API 返回原始分辨率，Workflow 使用 720px 压缩图，两者数据结构不一致 | **中** | 开发维护成本高，容易引入 bug |
| 3 | 无设备分辨率配置/校验机制，跨设备坐标不可移植 | **高** | Workflow 在不同设备运行时可能全部偏移 |
| 4 | 坐标空间无明确标注，用户不知道参数是哪个分辨率下的坐标 | **中** | 用户填写 Workflow 参数时容易出错 |
| 5 | 双重 `Math.round` 舍入导致 1-2px 累积误差 | **低** | 大部分场景可忽略 |
| 6 | Track 页面传输原始分辨率大图，带宽浪费 | **中** | 调试体验较差 |

## 4. 建议方案：设备分辨率强制约定

### 4.1 核心思路

通过 **配置文件** 强制约定设备的"标准分辨率"，所有坐标参数统一基于该分辨率，运行时自动适配。

```yaml
# device-profiles.yaml（示例）
devices:
  - id: "ldplayer-0"
    name: "雷电模拟器 0"
    standardResolution:
      width: 1920
      height: 1080
```

### 4.2 关键设计原则

1. **所有面向用户的坐标统一为"标准分辨率"坐标**
   - Track 页面框选输出 → 标准分辨率
   - Workflow 节点参数 → 标准分辨率
   - 日志回显 → 标准分辨率

2. **Server 端负责一切坐标转换**
   - 截图后获取实际分辨率
   - 计算 `实际分辨率 / 标准分辨率` 缩放因子
   - 所有识别操作在压缩图空间执行，结果转回标准分辨率

3. **设备分辨率不匹配时主动告警**
   - 如果实际分辨率与配置的标准分辨率比例不一致（如 16:9 vs 4:3），拒绝执行并提示

### 4.3 改造要点

#### Track 页面改造
- 截图 API 统一返回压缩图 + `originalWidth/originalHeight`（与 Workflow 一致）
- `useSelection.ts` 的 `scaleX/scaleY` 改为 `originalWidth / displayWidth`
- 修复 `saveSelection()` 裁切：在原始图上裁切，或先将坐标映射到压缩图

#### Workflow 改造
- 节点配置面板增加"基于标准分辨率"的提示文案
- engine.ts 的 `scaleRegionToCompressed` 改为 `标准分辨率 → 压缩图` 转换
- 增加分辨率不匹配的 warn 日志

#### 日志页面
- 无需大改，只需确保 `screenshot.originalWidth/Height` 是标准分辨率而非实际分辨率

### 4.4 坐标转换伪代码

```ts
// 配置
const standardWidth = 1920
const standardHeight = 1080

// 截图后
const actualWidth = screenshot.width   // 实际设备分辨率
const actualHeight = screenshot.height

// 用户坐标（标准分辨率）→ 实际分辨率
function standardToActual(x: number, y: number): [number, number] {
  return [
    Math.round(x * actualWidth / standardWidth),
    Math.round(y * actualHeight / standardHeight),
  ]
}

// 实际分辨率 → 压缩图空间
function actualToCompressed(x: number, y: number, compressedW: number, compressedH: number): [number, number] {
  return [
    Math.round(x * compressedW / actualWidth),
    Math.round(y * compressedH / actualHeight),
  ]
}

// 合并：标准分辨率 → 压缩图（一步到位，减少舍入误差）
function standardToCompressed(x: number, y: number, compressedW: number, compressedH: number): [number, number] {
  return [
    Math.round(x * compressedW / standardWidth),
    Math.round(y * compressedH / standardHeight),
  ]
}
```

## 6. 已实施改造（2026-07-12 ~ 07-14）

### 6.1 标准分辨率空间（2026-07-12）

**核心决策**：截图强制 resize 到标准分辨率宽度（最大边 1280px）。

配置文件 `server/automan.config.json5`：
```json5
{
  resolution: { width: 1280, height: 720 },
  screenshot: { interval: 2000 },
}
```

**ScreenshotDispatcher** 强制 `sharp.resize({ width: 1280 })`，截图空间 = 标准分辨率空间。

### 6.2 坐标转换系统（coordinate.ts）

引入 `ScaleFactor` 抽象，统一管理坐标缩放：

```ts
// 计算缩放因子（最长边 = 1280）
const sf = computeScaleFactor(actualWidth, actualHeight)
// sf = { scaleX, scaleY, standardWidth, standardHeight }

// 标准空间 → 实际设备坐标
const [ax, ay] = toActualPoint(x, y, sf)        // 单点
const actual = toActualRegion(region, sf)        // 区域

// 实际设备 → 标准空间
const [sx, sy] = toStandardPoint(x, y, sf)
```

### 6.3 screencap 分辨率替代 wm size（2026-07-14）

**问题根因**：`adb shell wm size` 始终返回设备**自然方向（竖屏）**的分辨率，
但 `screencap` 截取的是**当前屏幕方向**的像素。当游戏横屏运行时：

| 来源 | 真机（横屏游戏） | 模拟器 |
|------|----------------|--------|
| `wm size` | 1440×3168（竖屏） | 720×1280（竖屏） |
| `screencap` | 3168×1440（横屏） | 1280×720（横屏） |

宽高互换导致 `computeScaleFactor` 算出竖屏标准空间，与横屏截图坐标不匹配。

**修复**：新增 `adbService.getScreencapSize()`，直接读取 screencap PNG 头部
（bytes 16-23）获取实际像素方向的宽高，无需完整解码图片。

**获取分辨率优先级**：
1. `getScreencapSize()` — 首选，方向与截图一致
2. `getScreenSize()`（wm size）— 回退方案
3. 配置默认值 — 最终回退

**模拟器不受影响**：scale ≈ 1.0，宽高互换无实际偏差。

### 改造前后对比

| 环节 | 改造前 | 改造后 |
|------|--------|--------|
| 截图空间 | 720px 压缩图 | 1280px 标准分辨率（最长边） |
| 分辨率来源 | `wm size`（竖屏方向） | `screencap` PNG 头部（实际方向） |
| 坐标转换 | 无统一抽象 | `ScaleFactor` + `toActualPoint/Region` |
| Engine 坐标链路 | `scaleRegionToCompressed` | 已删除，标准空间直接使用 |
| 累积舍入误差 | 1-2px | 已消除 |
| 横屏真机支持 | 坐标偏移严重 | 正确 |

### 改造后的坐标链路

```
用户坐标（标准空间，最长边 1280）
    ↓ clampPoint / clampRegion 限制在截图尺寸内
标准空间坐标
    ↓ toActualPoint / toActualRegion（× scaleFactor）
设备实际像素坐标
    ↓ adb shell input tap x y
设备屏幕
```

### 日志输出示例

点击/范围点击节点执行时输出完整坐标转换链路：
```
点击: 标准(450,320) → 实际(1114,792)
范围点击: 标准[450,320,500,370] → clamp[450,320,500,370] → 实际[1114,792,1238,919] (截图1280x581)
```

设备连接时输出分辨率信息：
```
设备分辨率 (screencap): 3168x1440

```

### 日志回显标注盒尺寸

注解盒尺寸根据实际数据动态计算：

| 注解类型 | 盒尺寸来源 |
|---------|----------|
| 识图 (bbox) | `templateW × templateH`（从模板图片 PNG 头部解析） |
| 识字/找字 (text) | `m.w × m.h`（OCR 返回的实际文字块尺寸） |
| 范围点击 (area) | `region[x1,y1,x2,y2]`（区域本身） |
| 点击 (click) | 固定波纹特效 |
