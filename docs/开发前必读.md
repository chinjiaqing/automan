# 开发前必读

> 本文档是 Automan 项目的基础开发约束，所有参与开发的同学请在动手写代码之前仔细阅读。

---

## 1. 语言规范

- **严格使用 TypeScript**，禁止使用 `.js` 文件（配置文件除外）
- 所有模块、函数、变量必须有明确的类型标注，禁止滥用 `any`
- 如确实无法确定类型，使用 `unknown` 并在使用时做类型收窄

---

## 2. 项目目录划分

| 目录 | 职责 |
|------|------|
| `server/` | 服务端代码（Fastify 应用、WS 通信、设备层、数据库等） |
| `web/` | 前端 UI 代码（设备管理、开发板工具等） |
| `shared/` | 前后端共用代码（公共 type 定义、常量、工具函数等） |
| `docs/` | 文档（产品文档、开发总结等） |

- `shared/` 中的代码**不得依赖** `server/` 或 `web/` 中的任何模块
- `server/` 和 `web/` 可以引用 `shared/`，但**不得互相引用**

---

## 3. 模块系统

- 使用 **ESM（`"type": "module"`）**，所有 import/export 使用 ESM 语法
- 禁止使用 `require()` / `module.exports`
- 导入路径必须带文件扩展名（如 `import { foo } from './foo.js'`），tsx 运行时要求如此

---

## 4. 代码风格

- 统一使用 **Prettier** 格式化代码，提交前必须格式化
- 变量 / 函数命名使用 `camelCase`
- 类 / 接口 / 类型命名使用 `PascalCase`
- 常量使用 `UPPER_SNAKE_CASE`
- 单个文件职责单一，不宜超过 300 行，超出应拆分

---

## 5. WebSocket 开发约束

- **禁止在 route 中直接写 WebSocket 逻辑**
- 必须严格分层：
  - `ws.gateway.ts` — 连接管理
  - `ws.dispatcher.ts` — 消息分发
- 所有 WS 消息类型必须在 `shared/src/types.ts` 中定义（`WsMessageType` 枚举），前后端共用
- 前端 WS 封装在 `web/src/composables/useWebSocket.ts`，单例模式，15s ping 保活

---

## 6. Actor 系统约束

- 每个 Task Actor **必须状态隔离**，禁止通过全局变量共享状态
- Actor 之间如需通信，必须通过 ActorManager 或 EventBus 中转
- Actor 必须实现完整的生命周期：`start()` / `stop()` / `receive(msg)`
- **禁止在 Actor 中使用 `while(true)` 阻塞循环**，必须使用 `await sleep()` 或队列调度

---

## 7. 异步与错误处理

- 所有异步操作必须使用 `async/await`，禁止裸 `.then()` 链
- 所有 `async` 函数必须有 `try/catch` 或在调用处处理异常
- 禁止吞掉错误（空 catch 块），至少要打印日志

---

## 8. 日志规范

- 使用统一的 Logger 封装（`core/logger.ts`），禁止直接 `console.log`
- Actor 产生的日志必须通过 EventBus 回传，供前端实时展示
- 日志格式建议：`[模块名] [级别] 消息内容`

---

## 9. Device 层约束

- Actor **不得直接操作设备**（截图 / 点击 / 识别），必须通过 Device Layer 调用
- Device 层是所有硬件 / 外部交互的唯一入口，方便未来替换实现

---

## 10. 提交规范

- 提交前必须通过 TypeScript 类型检查（`tsc --noEmit`）
- 提交信息格式建议：`<type>: <description>`，如 `feat: add actor manager`、`fix: ws disconnect handling`
- 常用 type：`feat` / `fix` / `refactor` / `docs` / `chore` / `style`

---

## 11. 包管理

- **统一使用 `pnpm`** 作为包管理器，禁止使用 `npm` / `yarn`
- 安装依赖：`pnpm install`
- 添加依赖：`pnpm add <package>`
- 添加开发依赖：`pnpm add -D <package>`
- 移除依赖：`pnpm remove <package>`
- 禁止手动修改 `pnpm-lock.yaml`，锁文件由 pnpm 自动维护

---

## 12. 工作汇报

- 每次工作完成后，必须在 `docs/memory` 目录下撰写工作总结
- 文件名格式：`YYYYMMDDHHmmss.md`（如 `20260708180000.md`）
- 总结内容应包含：本次完成的事项、变更的文件、关键决策及原因、遗留问题或下一步计划
- 工作总结由执行者（人或 AI）在完成工作时主动撰写，不需额外提醒

---

## 13. 前端开发约束

- **样式统一使用 UnoCSS**，禁止内联 `style` 属性（动态计算除外）
- **图标统一使用 primeicons CSS 类**（`<i class="pi pi-xxx" />`），禁止使用 SVG 图标或其他图标库
- **PrimeVue 仅使用 Dialog 组件**，其余 UI 用原生 HTML + UnoCSS shortcuts
- 前端分层：`views/` → `components/` → `composables/` → `api/`
- 全局状态使用 composable 单例模式（模块级 ref），不使用 Pinia/Vuex
- HTTP API 统一在 `api/index.ts` 封装，业务组件不直接调用 fetch

---

## 14. API 开发约束

- API 路由**只使用 GET / POST**，不使用 PUT / DELETE
- 所有 API 响应必须使用统一格式：`ApiResponse<T>` / `ApiError`（定义在 `shared/types.ts`）
- 前端 HTTP 封装在 `api/index.ts` 的 `request()` 函数中统一处理网络异常和 HTTP 状态
- 新增 API 必须同时更新 `shared/types.ts` 中的请求/响应类型
